# AI Newsletter Curator — 프로젝트 기획서

> 흩어진 웹 콘텐츠를 AI로 요약해 관심사에 맞춰 전달하는 개인 맞춤 큐레이션 서비스
> *AI-powered newsletter curator that collects, summarizes, and delivers a personalized digest to your inbox.*

- **버전**: v1.1 (아키텍처/스택 개선 반영)
- **레포명 후보**: `sift` · `lede` · `skim` (확정 후 교체)
- **문서 성격**: 백엔드 포트폴리오용 프로젝트 기획서

---

## 1. 프로젝트 개요

### 한 줄 소개
다양한 뉴스레터·블로그·기술 아티클을 자동 수집하고 AI로 요약하여, 사용자 관심사에 맞춰 전달하는 개인 맞춤형 정보 큐레이션 플랫폼.

### 해결하려는 문제
개발자와 직장인은 수많은 뉴스레터·콘텐츠를 구독하지만 실제로 읽는 비율은 매우 낮다. 본 프로젝트는 다음을 목표로 한다.

- 흩어진 정보 소스 통합
- 긴 글에 대한 빠른 이해(요약)
- 관심사 기반 콘텐츠 필터링
- 원하는 채널로 자동 전달

### 포트폴리오 관점 목표
백엔드 개발자로서 다음 역량을 보여주는 것을 목표로 한다.

- 서버 아키텍처 설계 (API 서버 / 워커 / 스케줄러 분리)
- 데이터 수집 파이프라인 구축
- AI 활용 서비스 개발 (LLM 추상화·비용 최적화)
- 비동기/백그라운드 작업 처리
- 추천 로직 설계
- 컨테이너 기반 운영 환경 구축

## 2. 핵심 기능

### 2.1 콘텐츠 수집

**지원 예정 소스**: 네이버 블로그, 티스토리, 브런치, Velog, Medium, 그 외 RSS 제공 사이트

**수집 방식**
- **RSS 우선 수집** — 위 소스 대부분이 RSS를 제공하므로 1차 버전은 RSS만으로 커버한다.
- RSS 미지원/약한 사이트는 크롤링으로 확장 (robots.txt·이용약관 준수, 요청 간격 제한).
- APScheduler로 주기적 스케줄링 (운영 단계에서 별도 워커로 분리 가능).

**수집 정보**: 제목, 작성자, 본문, 작성일, 태그, 원본 링크

### 2.2 AI 요약

수집된 콘텐츠를 LLM으로 요약한다.

**생성 정보**
- 3줄 요약
- 핵심 키워드 (예: `Flask`, `Celery`, `Redis`, `Background Job`)
- 예상 읽기 시간

**요약 설계 원칙**
- **LLM 호출은 인터페이스로 추상화**하여 모델을 쉽게 교체할 수 있게 한다.
- 요약·키워드는 **저비용 모델(flash/mini/nano급)**로 시작하고, 품질 차이를 측정한 경우에만 상위 모델로 승급한다.
- **프롬프트 캐싱**을 활용해 반복 시스템 프롬프트 비용을 절감한다.
- **예상 읽기 시간은 LLM이 아니라** 본문 글자 수 ÷ 분당 읽기 속도로 직접 계산한다(토큰 절약).
- 출력은 **구조화(JSON)** 형식으로 받아 파싱한다.

**요약 예시**
```
- Flask 비동기 작업 처리 방법 소개
- Celery와 Redis 활용 사례 설명
- 운영 환경 적용 시 고려사항 정리
키워드: Flask, Celery, Redis, Background Job
예상 읽기 시간: 5분
```

### 2.3 관심사 기반 필터링

사용자는 관심사를 등록할 수 있다. (예: `Python`, `Flask`, `AWS`, `AI`, `Docker`)

수집된 콘텐츠의 키워드와 사용자 관심사를 비교해 **관심도 점수**를 계산하고, 사용자별 맞춤 콘텐츠를 선정한다.

> **MVP 구현**: 복잡한 모델 없이 *관심 키워드 ↔ 콘텐츠 키워드 겹침 점수* 같은 단순 규칙 기반으로 시작한다.

### 2.4 개인 맞춤 추천

사용자의 행동 데이터를 기반으로 추천한다.

**추천 기준**: 조회 기록, 클릭 기록, 저장 기록, 관심 키워드

> 행동 데이터를 쌓기 위해 **이벤트 로그 테이블(`user_events`)**을 도입한다(5장 참고). 단순 점수 기반 추천 → 이후 벡터 검색/개인화 엔진(8장 고도화)으로 발전시킨다.

**추천 예시**: 최근 AWS 관련 글을 자주 읽은 사용자에게 AWS Lambda·ECS·CloudWatch 관련 콘텐츠를 우선 추천.

### 2.5 알림 시스템

사용자가 원하는 채널로 요약 결과를 전달한다.

**지원 예정**: Email(MVP) → Slack, Discord, Telegram(고도화)

> 알림은 건별 발송이 아니라 **하루치 다이제스트 한 통**으로 묶어 발송한다. 이메일 발송은 AWS SES 등 트랜잭션 메일 서비스 사용.

**전달 예시**
```
[오늘의 추천 콘텐츠]

1. Flask 비동기 처리 완벽 가이드
   - 3줄 요약 ...
   - 예상 읽기 시간 5분

2. AWS ECS 운영 노하우
   - 3줄 요약 ...
   - 예상 읽기 시간 7분
```

## 3. 시스템 아키텍처

API 서버와 무거운 파이프라인 작업을 **분리**하는 것이 핵심이다.

```
            ┌─────────────┐
            │  Scheduler  │  (APScheduler / Celery Beat)
            └──────┬──────┘
                   ▼  (작업 등록)
            ┌─────────────┐
            │   Worker    │  (Celery / RQ / Arq)
            └──────┬──────┘
                   ▼
  Crawler → Raw Content DB → AI Summarizer → Keyword Extractor
                                                   │
                                                   ▼
                                      Recommendation Engine
                                                   │
                                                   ▼
                                       Notification Service → User

  ┌─────────────┐
  │ API Server  │  (Flask 또는 FastAPI) — 사용자/관심사/관리자 API
  └─────────────┘
```

**역할 분리**
- **API 서버**: 사용자·관심사·관리자 요청 처리
- **워커**: 크롤링·요약 등 무거운 백그라운드 작업
- **스케줄러**: 수집·발송 주기 트리거

## 4. 기술 스택

| 영역 | 선택 | 비고 |
|------|------|------|
| Language | **Python 3.12+** | 기존 3.7은 지원 종료(EOL) → 최신 LTS급으로 상향 |
| Web Framework | Flask 또는 **FastAPI** | FastAPI는 async·자동 OpenAPI 문서·Pydantic 검증 이점 |
| Background Worker | Celery / RQ / Arq | 파이프라인을 API와 분리 |
| Scheduler | APScheduler (→ Celery Beat) | MVP는 APScheduler로 충분 |
| Database | PostgreSQL | |
| Cache / Broker | Redis | 캐시 + 작업 큐 브로커 |
| AI | **LLM 추상화 레이어** | 저비용 모델로 시작, 프롬프트 캐싱 활용, 공급자 교체 가능하게 설계 |
| Container | Docker / docker-compose | app + postgres + redis + worker 일괄 구동 |
| Infra | AWS EC2 / RDS / S3 | 아키텍처 기준. 상시 가동 비용은 운영 정책에서 검토 |

## 5. 데이터베이스 설계

### users — 사용자 정보
`id`, `email`, `nickname`, `created_at`

### sources — 수집 대상
`id`, `source_name`, `source_url`, `source_type`

### contents — 원문 데이터
`id`, `source_id`, `title`, `content`, `original_url`, `published_at`, `content_hash`
> `content_hash`로 중복 콘텐츠를 차단하고, 요약 캐싱의 키로 사용.

### summaries — AI 요약 데이터
`id`, `content_id`, `summary`, `keywords`, `reading_time`
> 본문 해시 기준으로 캐싱하여 같은 글을 두 번 요약하지 않는다.

### user_interests — 사용자 관심사
`id`, `user_id`, `keyword`

### user_events — 사용자 행동 로그 *(신규 추가)*
`id`, `user_id`, `content_id`, `event_type`(view/click/save), `created_at`
> 추천 엔진의 입력 데이터. 기존 설계에 누락되어 있던 부분을 보완.

### notifications — 발송 이력
`id`, `user_id`, `content_id`, `send_type`, `status`

## 6. 핵심 기술 포인트 (포트폴리오)

- **백엔드**: 크롤링 파이프라인, API 설계, DB 모델링, 비동기 처리
- **데이터 처리**: 중복 콘텐츠 제거(URL 정규화 + 본문 해시), 요약 캐싱, 키워드 추출
- **AI 활용**: 자동 요약, 키워드 생성, LLM 추상화·비용 최적화, 단순 추천 로직
- **운영**: docker-compose 기반 재현 환경, 로그/모니터링, 장애 복구, 비용 최적화

## 7. 개발 순서 (빌드 전략)

기능을 가로로 쌓지 않고 **하나의 경로를 끝까지 관통하는 수직 슬라이스**부터 시작한다.

### 0단계 — 수직 슬라이스 (약 1주)
1. RSS 피드 **하나** 수집
2. DB에 원문 저장
3. LLM으로 3줄 요약 1건 생성
4. 결과를 **내 이메일로** 발송

→ 수집 → 요약 → 전달 파이프라인 뼈대 검증.

### MVP (1차 버전)
- RSS 기반 다중 소스 수집 + 중복 제거
- LLM 요약(캐싱 적용)
- 사용자 관심사 등록 + 단순 점수 기반 필터링
- Email 다이제스트 발송
- 관리자 페이지
- **예상 개발 기간**: 약 2~4주

## 9. 프로젝트 차별점

단순한 뉴스 수집기가 아닌 **"사용자 관심사 기반 AI 뉴스 큐레이션 플랫폼"**을 목표로 한다.

> **수집 → 요약 → 추천 → 전달** 전 과정을 자동화하여 정보 과부하 문제를 해결한다.
